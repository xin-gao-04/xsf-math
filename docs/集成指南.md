# xsf-math 集成指南

## 1. 集成方式概览

`xsf-math` 当前是一个基于 CMake 的头文件库。

推荐的接入方式有两种：

- 方式 A：把本仓库作为源码子目录，通过 `add_subdirectory` 接入上层工程。
- 方式 B：仅拷贝 `include/xsf_math` 并在上层工程中追加头文件搜索路径。

如果上层工程已经使用 CMake，优先采用方式 A，这样能顺便复用本仓库的示例与测试配置。

## 2. CMake 接入

上层工程可直接这样使用：

```cmake
add_subdirectory(path/to/xsf-math)

target_link_libraries(my_target PRIVATE xsf_math)
```

由于 `xsf_math` 是 `INTERFACE` 目标，链接动作本质上是在继承头文件搜索路径和编译选项。

## 3. 头文件使用入口

按使用习惯有两种入口：

- 聚合入口：`#include <xsf_math/xsf_math.hpp>`
- 按模块精确包含，例如：
  - `#include <xsf_math/core/coordinate_transform.hpp>`
  - `#include <xsf_math/radar/radar_equation.hpp>`
  - `#include <xsf_math/guidance/proportional_nav.hpp>`

如果上层只需要少量能力，建议按模块精确包含，便于控制编译依赖和阅读边界。

## 4. 推荐的最小使用面

如果下游只需要：

- 基础数学、常量、插值：从 `core` 开始
- 雷达探测链路：从 `radar` 开始
- 跟踪滤波与关联：从 `tracking` 开始
- 制导/气动/引信/杀伤：从 `guidance`、`aero`、`lethality` 开始
- 轨道传播：从 `orbital` 开始

不要默认包含整个 `xsf_math.hpp`，除非应用确实需要跨模块联动。

## 5. 示例构建

仓库默认会编译示例和测试：

```powershell
cmake -S . -B build
cmake --build build --config Release
ctest --test-dir build -C Release --output-on-failure
```

对应入口如下：

- 雷达链路示例：`examples/radar_detection_example.cpp`
- 导弹交战示例：`examples/missile_engagement_example.cpp`
- 轨道传播示例：`examples/orbital_example.cpp`

## 6. 当前构建注意点

- 项目使用 C++17。
- 在 MSVC 下显式启用了 `/utf-8`，以支持中文注释和文档字符串的正常编译。
- 当前测试覆盖核心模块、雷达模块以及制导/气动/电子战/杀伤/轨道联合能力。

## 7. 不建议依赖的假设

当前不建议下游默认依赖以下假设：

- 存在稳定的安装包布局
- 已发布独立二进制 SDK
- 所有模块都会长期保持完全不变的聚合头入口

更稳妥的做法是把本仓库视为源码级数学能力库来集成。
