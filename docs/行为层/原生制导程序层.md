# 原生制导程序层

本文档描述 `include/xsf_behavior/guidance_programs.hpp` 这一层的定位。

它的目标不是重新设计一套飞行控制业务逻辑，而是把 `xsf-core` 中 `GuidanceProgram / GuidanceComputer`
里最核心、最可独立复用的程序级流程拆出来，变成可被外部框架直接调用的原子单元。

## 1. 当前已迁入的程序

- `legacy_guidance_program`
- `altitude_guidance_program`
- `intercept_guidance_program`
- `legacy_flight_path_angle_program`
- `gravity_bias_program`
- `gravity_turn_program`

## 2. 这一层和原子控制器层的区别

原生制导程序层保留的是 `xsf-core` 的程序级思想：

- LOS 导引先算 `pursuit / PN / APN`
- 再叠加 `angle_offset`
- 再按需要做时间常数平滑
- 再处理 `flight_path_angle / altitude`
- 最后叠加 `gravity_bias / lateral_bias`

原子控制器层保留的是对外更好接入的动作接口：

- 拉升
- 转弯
- 定高
- 航点跟踪
- 进近
- 拉平

所以：

- 如果你想尽量复现 `xsf-core` 的程序级执行顺序，用原生制导程序层
- 如果你想直接把某个飞行动作托管给外部仿真框架，用原子控制器层

## 3. 当前公共数据

程序层统一依赖：

- `guidance_program_state`
- `guidance_phase_options`
- `guidance_program_commands`
- `guidance_program_result`

其中 `guidance_phase_options` 对齐了 `xsf-core` 当前已经迁入的关键相位参数，例如：

- `pn_gain`
- `vp_gain`
- `cos_los_offset`
- `gee_bias`
- `lateral_gee_bias`
- `max_gee_cmd_mps2`
- `max_pitch_angle_rad`
- `max_ascent_rate_mps`
- `max_descent_rate_mps`

## 4. 当前已保留的 xsf-core 思路

### `legacy_guidance_program`

保留了最典型的 `xsf-core` 执行链：

1. 根据 LOS 与 `cos_los_offset` 判断使用 `pursuit` 还是 `PN/APN`
2. 叠加 `angle_offset`
3. 按 `time_constant` 平滑横纵向指令
4. 若存在 `commanded_flight_path_angle`，覆盖纵向指令
5. 若存在 `commanded_altitude`，覆盖纵向指令
6. 若纵向程序未覆盖，则叠加 `gravity_bias`
7. 最后叠加 `lateral_bias`

### `intercept_guidance_program`

保留拦截程序里最关键的切换逻辑：

- `pursuit_nav_gain`
- `pro_nav_gain`
- `cos_switch_angle`
- `pn_method`

### `altitude_guidance_program`

保留 `AltitudeGuidance` 的核心公式：

- 高度差先换算为所需垂向速度
- 再换算为飞行路径角
- 再换算为完成该角度变化的纵向加速度
- 同时支持 `AGL/MSL` 语义

### `legacy_flight_path_angle_program`

保留旧式飞行路径角程序的核心行为：

- 根据目标飞行路径角输出纵向加速度
- 使用与 `xsf-core` 相同的完成判据

### `gravity_bias_program`

单独暴露 `ApplyGravityBias` 这一类能力，便于外部框架在不跑完整 legacy program 的情况下也能复用。

### `gravity_turn_program`

保留 `xsf-core` 中“纵向不主动给额外俯仰控制，主要依赖重力转弯”的程序语义。

## 5. 当前仍未迁完的程序精髓

这一层还没有覆盖完整的 `xsf-core` 程序族，至少还缺：

- `attitude_program`
- 新式 `flight_path_angle_program`
- `orbit_insertion_program`
- 更完整的程序配置输入优先级链
- 与 mover 命令接口更贴近的速度/推力/最小离地高度约束

所以现在的状态是：

- 已经从“只有动作控制器”推进到“有第一批原生 guidance program”
- 但离“完整移植 `xsf-core` 精髓”还有后续批次要补

## 6. 推荐使用方式

如果你的外部框架已经有对象模型和状态推进，推荐这样接：

1. 把平台状态适配成 `guidance_program_state`
2. 把阶段参数适配成 `guidance_phase_options`
3. 选择一个 guidance program 执行
4. 直接消费 `accel_cmd_ecs / angle_rate_cmd_rad_s`
5. 再由外部框架把这些命令映射到底层 mover 或动力学模型
