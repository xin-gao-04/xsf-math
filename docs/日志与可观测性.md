# 日志与可观测性

> 对应 `include/xsf_common/log_config.hpp` 和 `include/xsf_common/log.hpp`。

## 1. 设计目标

仿真代码有一类特殊的调试需求：运行结果正确，但中间过程不透明。我们不仅想知道"最终 Pk 是多少"，还想知道"末段进入时的航迹角速率是多少"、"每步的 SNR 下降到什么时候过门限"。

日志层解决的正是这个问题：

- 在仿真代码中标注关键状态和事件
- 不改变仿真逻辑和计算结果
- 按需开关，不影响发布版性能
- 为后续结构化事件和回放系统保留扩展接口

## 2. 快速上手

### 2.1 在代码中使用

```cpp
#include <xsf_common/log.hpp>

// 场景初始化
XSF_LOG_INFO("missile scenario: range={:.0f}m  TTI={:.1f}s", range, tti);

// 每步跟踪（高频，默认 DEBUG 级别）
XSF_LOG_DEBUG("step {}: accel={:.1f}g  alt={:.0f}m", step, accel_g, alt);

// 关键事件（TRACE，仅需极细粒度排查时启用）
XSF_LOG_TRACE("LOS rate = {:.4f} rad/s", omega.magnitude());

// 异常参数
XSF_LOG_WARN("dynamic pressure {:.0f} Pa below threshold", q);

// 不可能发生的条件
XSF_LOG_ERROR("nav_ratio <= 0 is invalid: {}", nav_ratio);
```

格式化语法与 Python `f-string` 类似，由 spdlog 内置的 fmtlib 提供。

### 2.2 运行时控制

```cpp
// 在 main() 或仿真初始化中调用一次
xsf::log::set_level(xsf::log::level::debug);     // 运行时改为 DEBUG
xsf::log::set_level(xsf::log::level::info);      // 只看关键事件
xsf::log::set_pattern("[%T] [%l] %v");            // 自定义格式
```

不调用任何初始化函数也能正常工作——日志器第一次被使用时自动以默认配置创建。

## 3. 级别设计指南

| 级别 | 宏 | 适用场景 |
|------|-----|---------|
| TRACE | `XSF_LOG_TRACE` | 每帧/每步的中间计算值；用于排查单个算法的数值行为。默认编译关闭 |
| DEBUG | `XSF_LOG_DEBUG` | 每 0.5-1s 一次的状态摘要；制导步长、SNR 变化、过载数值 |
| INFO  | `XSF_LOG_INFO`  | 场景开始/结束；关键状态迁移（末段进入、引信触发、轨道宣告） |
| WARN  | `XSF_LOG_WARN`  | 可疑但可继续的参数；能量不足；异常几何 |
| ERROR | `XSF_LOG_ERROR` | 非法输入；断言式的保护条件；不应到达的分支 |

## 4. 编译时开关

### 默认行为

| 构建类型 | 默认级别 |
|---------|---------|
| Debug（无 NDEBUG） | DEBUG（1）编译进去 |
| Release（有 NDEBUG） | INFO（2）编译进去 |

TRACE 和 DEBUG 在 Release 构建中被编译器在预处理阶段删除，零运行时开销。

### 强制指定级别

在任何包含 `log.hpp` 的文件之前定义：

```cpp
#define XSF_ACTIVE_LOG_LEVEL 0   // 强制 TRACE（最详细）
#define XSF_ACTIVE_LOG_LEVEL 6   // 强制 OFF（全部关闭）
#include <xsf_common/log.hpp>
```

或在 CMake 中：

```cmake
target_compile_definitions(my_target PRIVATE XSF_ACTIVE_LOG_LEVEL=0)
```

### 直接免安装启用日志

如果不想预先安装任何依赖，可以直接让 CMake 在配置阶段自动拉取 header-only `spdlog`：

```powershell
cmake -S . -B build -DXSF_LOGGING_PROVIDER=fetch
```

### 使用已有系统包

如果你的环境已经能通过 `find_package(spdlog)` 找到 `spdlog`，可以强制走系统包模式：

```powershell
cmake -S . -B build -DXSF_LOGGING_PROVIDER=system
```

### 自动模式

默认就是自动模式：

```powershell
cmake -S . -B build
```

它会先尝试系统包；找不到时再自动拉取 header-only `spdlog`。

### 完全禁用（无 spdlog 依赖）

```cmake
cmake -DXSF_ENABLE_LOGGING=OFF ..
```

此时所有 `XSF_LOG_*` 宏展开为 `(void)0`，连 `#include <spdlog/...>` 都不出现在编译单元中。

## 5. 为什么不直接用 spdlog

直接在算法头文件中 `#include <spdlog/spdlog.h>` 会带来两个问题：

1. **spdlog 的具体 API 泄漏进所有引用方**。消费者被迫了解 spdlog 的配置方式，更换后端需要改动所有调用点。
2. **一致性消失**。不同文件的日志器名称可能不同，格式不统一，级别无法统一控制。

`xsf_common/log.hpp` 把这两个问题解决了：所有日志都经过同一个 `"xsf"` 命名的日志器，调用方只看到 `XSF_LOG_*` 宏，后端可以无缝替换。

## 6. `vcpkg.json` 是做什么的

`vcpkg.json` 是当前仓库的依赖清单文件，用来声明这个项目在构建时需要哪些第三方库。

在本仓库里它现在主要承担两件事：

1. 声明 `spdlog` 是日志后端依赖。
2. 让使用 vcpkg manifest 模式的环境在配置 CMake 时自动解析依赖。

当前文件内容很简单：

```json
{
  "name": "xsf-math",
  "version": "0.1.0",
  "description": "Header-only math/simulation library for defense and space modeling",
  "dependencies": [
    "spdlog"
  ]
}
```

如果你的 CMake 已经接入 vcpkg toolchain，那么配置项目时会按这份清单去找 `spdlog`。  
如果没有接入 vcpkg，也可以直接使用 `XSF_LOGGING_PROVIDER=fetch` 自动拉取 header-only `spdlog`，不需要先安装。

```powershell
cmake -S . -B build -DXSF_LOGGING_PROVIDER=fetch
```

当前自动拉取时使用的 `spdlog` 版本由 `XSF_SPDLOG_GIT_TAG` 控制，默认值是：

```cmake
v1.17.0
```

可以按需覆盖：

```powershell
cmake -S . -B build -DXSF_LOGGING_PROVIDER=fetch -DXSF_SPDLOG_GIT_TAG=v1.17.0
```

## 7. 重要工程约束

### 总入口不要强制绑定日志

`xsf_math.hpp` 和 `xsf_behavior.hpp` 仍然不直接包含 `log.hpp`。  
这样外部使用者如果只想拿算法接口，不会因为总入口自动拉起日志配置。

### 关键头文件允许内置埋点

为了验证算法和行为逻辑，当前有少量关键头已经内置日志埋点，例如：

- `include/xsf_math/radar/radar_equation.hpp`
- `include/xsf_math/guidance/proportional_nav.hpp`
- `include/xsf_behavior/flight/basic_controllers.hpp`
- `include/xsf_behavior/flight/guidance_programs.hpp`

这些埋点统一经过 `xsf_common/log.hpp`，不开日志时自动退化为空操作。

### 不要在仿真主循环中输出 INFO 以上频率的日志

每帧或每步的高频输出应使用 TRACE 或 DEBUG。INFO 级别预期频率不超过每秒几条。

### 日志不得改变仿真结果

日志调用只能读状态，不能修改任何仿真变量。格式化参数中不允许出现副作用表达式。

## 8. 当前已覆盖的观测链

目前已经能直接看到三类内部过程：

- 探测链：单基地/双基地雷达方程、探测概率、探测距离
- 制导链：PN、APN、追踪导引、加速度限幅
- 飞行行为链：定高、定航向、进近、拉平、原生制导程序输出

推荐做法是：

1. 在示例或宿主程序里先调用 `xsf::log::set_level(...)`
2. 用 `INFO` 看阶段事件
3. 用 `DEBUG` 看控制输出摘要
4. 只在定位单个数值问题时开 `TRACE`

## 9. 与未来扩展的兼容性

### 9.1 结构化仿真事件

在基础日志之上可以增加一个结构化事件接口：

```cpp
// 未来可能的扩展（不破坏现有 XSF_LOG_* 调用）
xsf::event("fuze_trigger", {{"miss_m", miss}, {"pk", pk_val}, {"time_s", t}});
```

内部仍通过同一个 spdlog 日志器输出，格式化为 JSON 行；外部消费者可解析。

### 9.2 帧录制与回放

为 `"xsf"` 日志器附加一个自定义 sink（例如二进制环形缓冲区），不需要改动任何 `XSF_LOG_*` 调用：

```cpp
// 外部框架初始化时调用一次
auto replay_sink = std::make_shared<BinaryReplaySink>(...);
spdlog::get("xsf")->sinks().push_back(replay_sink);
```

### 9.3 外部仿真宿主

宿主在启动时调用 `xsf::log::set_level()` 和 `xsf::log::set_pattern()` 接管日志配置，或者用 `XSF_ENABLE_LOGGING=OFF` 完全关闭 xsf 日志并由宿主的日志系统接管。

### 9.4 行为树 / 状态机驱动

状态迁移事件使用固定前缀格式：

```cpp
XSF_LOG_INFO("[phase] {} -> {}", from_phase, to_phase);
```

后续结构化事件层可按前缀 `[phase]` 提取并路由，无需修改任何调用点。
