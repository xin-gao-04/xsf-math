# 基础知识整理

本文件对应算法层的基础支撑业务域，目的是在阅读 `core/` 代码之前，先统一最容易混淆的基础概念。

## 1. 这一层解决什么问题

基础支撑层不直接产出探测概率、制导过载或轨道机动结果。它解决的是更前面的三个问题：

- 物理常数和单位如何统一
- 三维几何量如何表达和变换
- 环境量如何从高度映射到密度、音速和动压

如果这些基础表达不统一，后面的雷达、制导、气动和轨道模块会各自形成一套局部坐标和局部单位，长期会导致结果不可组合。

## 2. 阅读代码前应先掌握的概念

### 2.1 向量、矩阵和坐标系

- `vec3`
  用于统一位置、速度、加速度、LOS 和法向量。
- `mat3`
  用于统一方向余弦矩阵和正交变换。
- `coordinate_transform`
  用于统一 WCS、ECS、ACS 等坐标系之间的映射。

当前库里最重要的不是“矩阵怎么乘”，而是“所有模块对坐标轴方向的约定是否一致”。如果坐标系约定漂移，后面的法向加速度、重力补偿和 LOS 角速度会整体错号。

### 2.2 大气模型

`atmosphere.hpp` 负责把高度映射为：

- 温度
- 压力
- 密度
- 音速
- 动压
- 马赫数

这些量是很多上层模块的共同输入：

- 雷达传播会用到大气条件
- 气动会直接用动压和马赫数
- 飞行行为会根据动压判断是否具备控制能力

### 2.3 插值与查表

工程计算里大量参数并不是封闭解析式，而是来自经验曲线、试验数据或离散表格。

`interpolation.hpp` 的价值不在于“提供一个线性插值函数”，而在于：

- 让查表和曲线拼接成为统一操作
- 为后续 Pk 表、气动曲线和 RCS 曲线提供一致接口

## 3. 与代码的对应关系

- `include/xsf_math/core/constants.hpp`
  提供常量和角度/单位换算基准。
- `include/xsf_math/core/vec3.hpp`
  提供三维向量基础运算。
- `include/xsf_math/core/mat3.hpp`
  提供三维旋转矩阵表达。
- `include/xsf_math/core/interpolation.hpp`
  提供工程曲线和查表插值基础。
- `include/xsf_math/core/coordinate_transform.hpp`
  提供姿态、方向和坐标系之间的转换。
- `include/xsf_math/core/atmosphere.hpp`
  提供 ISA 条件下的大气量和导出量。

## 4. 常见误区

### 4.1 俯仰角和航迹角混用

俯仰角描述机体朝向，航迹角描述速度方向。基础层不直接定义飞行动作，但坐标变换如果不区分这两个量，后续制导和飞行行为都会出错。

### 4.2 高度、地理坐标和局部坐标混用

同一个“位置”在工程里可能同时出现：

- 椭球坐标
- 地心坐标
- 局部东北天或北东地下
- 平台体轴坐标

基础层的任务就是把这些表达之间的关系固定下来，而不是在上层临时换算。

### 4.3 把动压当成附属量

动压不是“顺带算一下”的派生量。它是飞行能力、气动力和控制效果的核心量。很多上层算法最后是否可执行，都会回到动压这个量上。
